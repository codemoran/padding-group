## That sounds amazing, my CIO will love this, How do I use it??

We looked at the "REST" implementation of `api.left-pad.io` and then the "REST"
implementation of `right-pad.io/api`, and it was obvious they were doing it
completely wrong.

Why are they using the query string to pass in parameters, when you **SHOULD**
be using HTTP headers so Hackers cannot see your data.

Look at this URL, https://right-pad.io/api?str=hackers-can-see-this&len=68&ch=!
a hacker can see EXACTLY what you are trying to right-pad, do you want to be
responsible for Hackers having access to all of your data?

## Well than, what is the RIGHT way to do this?

You need to hide the data in the headers so hackers cannot see them.  We at
**The Padding Group** have been pioneers in leading the SecOps benefits of
**Header Driven Development**.

```
$ curl 'https://secure.rightpad.io' \
-H "X-STR: hackers-cannot-see-me-they-just-see-secure.rightpad.io" \
-H "X-LEN: 64" \
-H "X-CH: !"
{"str":"hackers-cannot-see-me-they-just-see-secure.rightpad.io!!!!!!!!!!"}
```

The header parameters names `X-STR`, `X-LEN`, and `X-CH` are the same as the
`left-pad` and `right-pad.io` just more secure, so swapping over to
rightpad.io should be a breeze.
